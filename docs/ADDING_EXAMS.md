# 問題・試験の追加ガイド（ADDING_EXAMS.md）

このドキュメントは、**どのAIモデル・どの作業者でも** CLOUDCERT_ サイトに新しい試験や問題を追加できるようにするための完全な手順書です。コードの変更は不要で、**データファイルの追加とマニフェストの1エントリ追記**だけで完了します。

---

## 全体構造

```
cloud-cert-quiz/
├── index.html              ← LP（変更不要）
├── exam.html               ← 演習プレイヤー（変更不要）
├── css/style.css           ← デザイン（変更不要）
├── js/
│   ├── main.js             ← LPロジック（変更不要）
│   ├── quiz.js             ← 演習エンジン（変更不要）
│   └── data/
│       ├── manifest.js     ★ 試験カタログ（試験追加時にここを編集）
│       └── exams/
│           └── <exam-id>.js ★ 問題データ（試験ごとに1ファイル作成）
└── docs/ADDING_EXAMS.md    ← このファイル
```

---

## 手順1: 問題データファイルを作成する

`js/data/exams/<exam-id>.js` を新規作成します。

- **exam-id の命名規則**: `<プロバイダ>-<試験コード小文字>`（例: `aws-saa-c03`, `gcp-ace`, `azure-az104`）
- **文字コード**: UTF-8（BOMなし）

### テンプレート

```js
// <試験の正式名称> 問題データ
window.CERT_EXAMS = window.CERT_EXAMS || {};
window.CERT_EXAMS["<exam-id>"] = {
  meta: {
    id: "<exam-id>",          // manifest.js の id と完全一致させる
    title: "<試験の英語正式名称>",
    code: "<試験コード>",      // 例: "SAA-C03"
    provider: "<aws|gcp|azure>"
  },
  questions: [
    {
      id: "q001",              // q001 から連番（3桁ゼロ埋め）
      type: "single",          // "single"=単一選択 / "multiple"=複数選択
      domain: "<出題ドメイン名>", // 本試験の公式ドメイン名（日本語）を使う
      difficulty: "medium",    // easy | medium | hard
      question: "問題文（日本語）",
      choices: [
        "選択肢1",
        "選択肢2",
        "選択肢3",
        "選択肢4"
        // multiple の場合は5択にしてもよい（最大6択 = A〜F）
      ],
      answer: [1],             // 正解の choices インデックス配列（0始まり）
                               // multiple なら複数指定: [0, 2]
      explanation: "解説文（日本語）"
    }
    // ... 以下、問題を追加
  ]
};
```

### フィールド仕様

| フィールド | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | string | ✔ | `q001` 形式の連番。試験内で一意 |
| `type` | string | ✔ | `"single"` または `"multiple"` |
| `domain` | string | ✔ | 本試験の公式出題ドメイン名。結果画面の「ドメイン別正答率」に使われるため、**同一ドメインは一字一句同じ表記**にすること |
| `difficulty` | string | ✔ | `easy` / `medium` / `hard` |
| `question` | string | ✔ | 問題文。`multiple` の場合は「〜を2つ選択してください」と選択数を明記 |
| `choices` | string[] | ✔ | 4〜6個。HTMLタグは使えない（エスケープされる） |
| `answer` | number[] | ✔ | 正解インデックス（0始まり）。`single` でも配列にする: `[2]` |
| `explanation` | string | ✔ | 正解の根拠＋**全誤答の誤りの理由**を含めること（後述の品質基準参照） |

## 手順2: manifest.js にエントリを追加する

`js/data/manifest.js` を開き、該当プロバイダ（aws / gcp / azure）の `exams` 配列にエントリを追加、または既存の `coming-soon` エントリを更新します。

```js
{
  id: "aws-dva-c02",            // 問題データの exam-id と完全一致
  code: "DVA-C02",
  title: "Developer – Associate",
  titleJa: "デベロッパー アソシエイト",
  level: "Associate",           // Foundational | Associate | Professional | Expert
  status: "available",          // ★ "coming-soon" → "available" に変更すると公開される
  questionCount: 30,            // ★ 実際の収録問題数に必ず更新する
  passLine: 72,                 // 合格ライン（%）。本試験の公式値を使う
  timeLimitMin: 130,            // 本試験の制限時間（分）※現状は表示のみ
  dataFile: "js/data/exams/aws-dva-c02.js",  // ★ 手順1で作成したファイルのパス
  description: "カードに表示される1行説明"
}
```

> **新しいプロバイダ（例: OCI）を追加する場合**: `providers` 配列に新オブジェクトを追加し、`css/style.css` の `:root` にプロバイダカラー変数（例: `--oci: #f80000;`）と、`index.html` のフィルタータブを1つ追加してください。

## 手順3: 動作確認

1. `index.html` をブラウザで開く（ローカルファイルのままでOK。fetchを使っていないため `file://` でも動作します）
2. 試験カードが `LIVE` バッジ付きで表示されることを確認
3. 「演習を開始」→ 練習モードで数問解く → 結果画面のドメイン別集計が正しいことを確認
4. 構文チェック（Node.jsがある場合）:
   ```
   node -e "global.window={}; require('./js/data/exams/<exam-id>.js'); const e=window.CERT_EXAMS['<exam-id>']; console.log(e.questions.length + ' questions OK');"
   ```

### よくあるミス

- `answer` のインデックスが1始まりになっている（**0始まり**が正しい）
- `manifest.js` の `id` と問題データの key が不一致 → 「問題データの読み込みに失敗」エラー
- `questionCount` の更新忘れ → LPの表示が実際と食い違う
- `domain` の表記ゆれ（例:「セキュアなアーキテクチャ設計」と「セキュアなアーキテクチャの設計」が混在）→ ドメイン集計が分裂する
- JS文字列内の `"` をエスケープし忘れる → 構文エラー

## 問題の品質基準（最重要・必読）

このサイトの価値は「**本試験と同レベルの問題**」であることです。以下の基準を満たさない問題は追加しないでください。

### ✔ 必須要件

1. **シナリオベースであること**
   - 「ある企業が／あるチームが〜という要件・制約を持っている。最も適切な（コスト効率が高い／運用負荷が少ない／セキュアな）ソリューションはどれか」という形式。
   - ❌ 禁止例: 「S3の最大オブジェクトサイズは？」「Lambdaのタイムアウト上限は？」のような単純知識問題。

2. **要件のトレードオフを含むこと**
   - 問題文に「最小限の運用負荷で」「コードの変更を最小限に」「最もコスト効率よく」などの**判断軸**を必ず入れる。正解はこの判断軸で決まる。

3. **誤答も実在する構成にすること**
   - 全選択肢が技術的に成立する構成で、誤答は「一見正しそうだが、要件のどこか1点を満たさない」ものにする。
   - ❌ 禁止例: 存在しないサービス名、明らかにデタラメな選択肢。

4. **解説は「正解の根拠」＋「全誤答の誤りの理由」をセットで**
   - 各誤答について「なぜその選択肢では要件を満たせないか」を具体的に書く。目安200字以上。
   - これにより1問で4つ分の学習効果を出すのがこのサイトのコンセプト。

5. **本試験の公式ドメイン構成に沿うこと**
   - 各試験の公式試験ガイドに記載された出題ドメインを `domain` に使い、ドメイン間の出題比率も公式比率に近づける。

### 出題ドメイン早見表（主要試験）

| 試験 | ドメイン |
|---|---|
| AWS SAA-C03 | セキュアなアーキテクチャの設計（30%）／弾力性に優れたアーキテクチャの設計（26%）／高パフォーマンスなアーキテクチャの設計（24%）／コストを最適化したアーキテクチャの設計（20%） |

### 推奨ワークフロー（AIモデルへの指示例）

新しい試験の問題を一括生成させる場合、このファイル全体を読ませた上で次のように指示します:

> `docs/ADDING_EXAMS.md` の規約と品質基準に完全準拠して、<試験名> の問題を<N>問作成し、`js/data/exams/<exam-id>.js` に保存してください。公式ドメイン比率に合わせて配分し、<N/5>問程度は type:"multiple" にしてください。完了後、manifest.js の該当エントリを status:"available"、questionCount:<N> に更新してください。

## 大量生成ワークフロー（100問以上 — parts + build-exam.js）

100問を超える一括生成では、1ファイルに直接書くと出力が長大になりミスが混入しやすい。
**パートファイル分割＋ビルドスクリプト**方式を使うこと（Node.js 必須）。

### 1. パートファイルを書く

`js/data/exams/parts/<exam-id>.part1.js`, `.part2.js`, … に 25〜40問ずつ分けて書く。
**`id` フィールドは書かない**（ビルド時に連番が自動採番される）:

```js
window.CERT_EXAM_PARTS = window.CERT_EXAM_PARTS || {};
(window.CERT_EXAM_PARTS["<exam-id>"] = window.CERT_EXAM_PARTS["<exam-id>"] || []).push(
  {
    type: "single",
    domain: "セキュアなアーキテクチャの設計",
    difficulty: "medium",
    question: "ある企業が…",
    choices: ["…", "…", "…", "…"],
    answer: [1],
    explanation: "正解の根拠…。Aは…のため誤り。Cは…。Dは…。"
  },
  // …続けて push
);
```

### 2. ビルド（検証＋最終ファイル生成）

```
node tools/build-exam.js <exam-id> "<英語タイトル>" "<試験コード>" <provider>
```

例: `node tools/build-exam.js aws-clf-c02 "AWS Certified Cloud Practitioner" "CLF-C02" aws`

スクリプトが全パートを結合し、型・選択肢数・answer範囲・解説の長さ・問題文重複などを
検証して `js/data/exams/<exam-id>.js` を生成する。**検証エラーが1件でもあれば生成されない**
（エラー一覧が出るのでパートファイルを修正して再実行）。
成功時はドメイン／type／難易度の分布サマリーが表示されるので、公式比率と照合すること。

### 3. 後始末

- manifest.js の該当エントリを更新（status / questionCount / dataFile）
- パートファイルは生成完了後に削除してよい（最終ファイルがマスターになる）

> ⚠️ **公開済み試験の再ビルドは禁止。** build-exam.js は id を1から振り直すため、
> 公開後に再ビルドするとユーザーの復習データと id の対応が壊れる。公開後の追記は
> 「既存試験に問題を追記する場合」（下記）の手順で最終ファイルに直接追加すること。

---

## 既存試験に問題を追記する場合

1. `js/data/exams/<exam-id>.js` の `questions` 配列の末尾に、連番を続けて追加（既存の `id` と重複しないこと）
2. `manifest.js` の `questionCount` を新しい合計値に更新
3. 既存問題の `domain` 表記と完全に一致させる（表記ゆれ厳禁）

> ⚠️ **既存問題の `id` は絶対に変更・再利用しないこと。**
> ユーザーの「間違えた問題の復習データ」（localStorage の `cloudcert:wrong:<exam-id>`）と挑戦履歴は問題 `id` に紐づいています。`id` を変更すると、ユーザーの復習リストが壊れます。問題を削除した場合は欠番のままにして構いません（エンジン側で現存しない `id` は自動的に無視されます）。

以上で完了です。コードの変更・ビルド・デプロイ作業は一切不要です（静的ファイルを置くだけ）。
